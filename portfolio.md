## 🤖 AI 에이전트 시스템 (AX Team + Agent Pipeline)

### 왜 했나

"AI 에이전트가 실제로 팀처럼 일할 수 있을까?"라는 질문에서 시작했다. 단순히 프롬프트를 체인으로 연결하는 게 아니라, 에이전트들이 서로의 발언에 반응하고, 충돌하고, 합의에 도달하는 구조를 만들고 싶었다.

AX Team은 그 실험이었고, Agent Pipeline은 AX Team에서 배운 구조를 적용해 실제로 코드를 찍어내는 공장으로 발전시켰다.

### 무엇을 했나

**AX Team** — 성향이 다른 AI 에이전트 6명이 토론·투표·합의를 거쳐 문서와 코드를 자동 생성하는 멀티 에이전트 협업 시뮬레이터

- 역할 기반(마케팅·기획) 대신 **성향 기반**(속도파·품질파·혁신파·실용파·아키텍처파·사용자파)으로 설계 → 같은 주제에 진짜 충돌 발생
- `classify_task()`로 태스크를 5가지 타입(build/feedback/review/discuss/plan)으로 분류 후 워크플로우 라우팅 → 고정 파이프라인 탈피
- `team_gate()`: 6명 전원 투표, 1명이라도 BLOCK이면 재작성
- Flask + Python generator + SSE로 에이전트 발언 실시간 스트리밍
- v1(CLI 3명) → v6(웹 6명 + 합의 체크리스트 + 코드 품질 개선)까지 6버전 반복 개선

**Agent Pipeline** — Planner → Builder → Critic → Fixer → Logger 에이전트 팀이 `tasks.json`의 태스크를 순서대로 완성된 Python 프로젝트로 만들어내는 자율 빌드 시스템

- Critic이 PASS를 줄 때까지 루프 무한 반복 → 품질 타협 없음
- `claude -p` subprocess로 매 호출마다 새 세션 → 에이전트 간 컨텍스트 오염 차단
- 태스크 소진 시 기존 목록 전체를 컨텍스트로 넘겨 새 아이디어 자동 생성 후 계속 실행
- `run.sh` 래퍼로 비정상 종료 시 자동 재시작, `tasks.json` 상태 기반 이어서 진행

### 어떻게 해결했나

**① 코드 잘림 문제**
LLM이 `max_tokens` 한도 내에 파일을 다 못 쓰고 함수 중간에서 조용히 절단되는 버그가 반복됐다. 에러도 없어서 저장 시점에는 알 수 없었다.
→ `is_truncated()`로 감지(식별자 중간 절단, 중괄호 불균형, 함수 선언 후 본문 없음 패턴) + 이어쓰기 최대 3회 루프로 해결.
→ 근본 원인이 토큰 예산 부족이었음을 발견 후, plan 단계 토큰 500→1500, 문서 생성 토큰 1500→8192로 상향.

**② 병렬 생성 시 import 불일치**
파일 여러 개를 동시에 생성하면 에이전트들이 서로 뭘 만드는지 모른 채 독립적으로 작성해 없는 경로·메서드를 참조하는 코드가 만들어졌다.
→ 합의 체크리스트를 생성 프롬프트에 삽입, 병렬 → 순차 실행으로 전환.

**③ 설명 텍스트 혼입**
에이전트가 코드 앞에 "설계 먼저 짚고 갈게요." 같은 설명을 붙여 파일에 그대로 저장됐다.
→ 마크다운 앞·뒤 텍스트 제거 전처리 추가, 마커 오염 제거 후처리 추가.

**④ 마법 딕셔너리 키 → dataclass 신호 타입**
제어 신호를 `{"__consensus__": ...}` 딕셔너리로 전달하다 오타로 묵묵히 실패하는 버그 반복.
→ `signals.py`에 `ConsensusSignal`, `GateResultSignal` dataclass 정의, `isinstance()` 체크로 명시적 분기.

**⑤ Rate limit 통합**
`doc_call()`과 `tool_agent_call()` 양쪽에 재시도 로직이 복붙되어 있었다.
→ `with_rate_limit_retry()` 함수로 통합, 병렬 호출 간 stagger 2초 + 최대 워커 3개로 rate limit 회피.

### 성과

- 실제로 6개 프로젝트(AI 면접 코치, 쿠팡 가격 모니터, K리그 해설, B2B SaaS, 테트리스, 할일 앱) 자동 생성하며 버그 패턴 발견 → 시스템 반복 개선
- AX Team v1 CLI → v6 웹 서비스까지 6단계 진화, 단위 테스트 12개 추가
- Agent Pipeline으로 20개 Python 에이전트 태스크 자동 빌드 진행 중

## 🛠️ OpsAgent — 도메인 적응형 내부 AI 코파일럿

### 왜 했나

기업 내부 데이터를 활용하는 AI 도구들이 대부분 단순 RAG 챗봇 수준이라는 문제를 느꼈다. "재고 위험 상품이 뭔지, 회의에서 어떤 결정을 했는지, 다음에 뭘 해야 하는지" — 이 세 가지를 한 번에 답하려면 문서 검색만으로는 부족하다. 정형 데이터(CSV)와 관계 정보(지식 그래프)를 결합해야 한다는 판단에서 시작했다.

### 무엇을 했나

- **3-Layer 채팅 라우터**: 질문을 data/doc/combined 3가지로 자동 분류 후 전문 엔진 위임
- **도메인 적응 구조**: 업종 프리셋 7개 + Claude 동적 분석으로 어떤 도메인이든 파일 올리면 즉시 작동
- **지식 그래프 4가지 빌드 경로**: 일반 문서(Claude entity/relation JSON 추출) / CSV(스키마 기반 2-pass) / Python(AST 분석) / 수동 입력
- **RAG 컬렉션 도메인별 분리**: 단일 컬렉션에 쌓으면 제조업 문서가 금융 질의에 검색되는 노이즈 문제 → `collection_name`으로 도메인마다 독립된 ChromaDB 컬렉션 사용
- **증거 기반 답변**: 모든 답변에 사용된 CSV·문서·KG 노드 수 badge 표시
- Streamlit으로 서비스화, 실제 배포

### 어떻게 해결했나

**도메인 무관성 문제**: 업종마다 entity 타입과 용어가 다르다. 고정 스키마로는 커버 불가 → 7개 업종 프리셋 + Claude 동적 분석으로 자동화, 모든 프롬프트에 `{domain_context}` 주입.

**RAG 노이즈 문제**: 단일 컬렉션 사용 시 도메인 간 검색 오염 → 도메인별 독립 컬렉션 분리로 정확도 개선.

**질문 유형에 따른 라우팅**: 숫자 기반 질문(CSV)과 맥락 기반 질문(문서)을 같은 엔진으로 처리하면 둘 다 답이 부정확해짐 → 라우터가 먼저 분류 후 전문 엔진 위임.

### 성과

- Streamlit Community Cloud 배포 및 운영 중
- 제조·금융·HR 등 업종 전환 즉시 작동하는 범용 구조 완성

## 📰 Newszips — 뉴스 자동 분류·요약 서비스

### 왜 했나

유튜브 뉴스 영상의 자막을 분류·요약해서 서비스로 만들어보고 싶었다. 기존 키워드 매칭 방식이 "삼성 화재"를 사회 뉴스로 분류하는 등 문맥을 못 읽는 문제를 보고, BERT fine-tuning으로 해결할 수 있다고 판단했다.

### 무엇을 했나

- **BERT fine-tuning**: klue/bert-base로 6개 카테고리(정치/경제/사회/문화/스포츠/IT) 분류 모델 학습
- **TF-IDF 카테고리별 학습**: 전체 corpus 기준 IDF 대신 카테고리 내부 IDF로 핵심어 추출 → 그 기사에서 실제로 특이한 단어 추출
- **LLM 요약 품질 개선**: 본문을 그대로 넘기지 않고 TF-IDF 핵심어를 먼저 추출해 프롬프트에 명시 → LLM이 집중할 포인트 유도
- **t-SNE 시각화**: BERT 임베딩 2D 시각화로 분류 결과 클러스터링 검증, 유사 기사 추천에 유클리드 거리 활용
- FastAPI 백엔드 + React 프론트엔드 풀스택 구현, Vercel + Supabase 배포

### 어떻게 해결했나

**분류 정확도 문제**: 키워드 매칭의 문맥 오류 → klue/bert-base fine-tuning으로 해결.

**요약 일관성 문제**: 본문을 그대로 LLM에 넘기면 문체·도입부에 끌려가 같은 사건도 요약이 매번 달라짐 → TF-IDF 핵심어 선추출 후 프롬프트에 삽입.

**TF-IDF 노이즈 문제**: 전체 corpus IDF 사용 시 "대통령"이 정치·스포츠 기사 모두에서 같은 가중치 → 카테고리별 독립 학습으로 해결.

### 성과

- **분류 정확도 96%** (klue/bert-base, 6개 카테고리)
- Vercel 프로덕션 배포 및 운영 중

## 🐶 Dog AI Agent — 강아지 품종 판별·맞춤 레시피 추천 (팀 프로젝트)

### 왜 했나

팀 프로젝트에서 Computer Vision 파이프라인을 end-to-end로 구현해보고 싶었다. 단순히 모델을 불러다 쓰는 수준을 넘어, 백본 비교 실험 → 성능 개선 → 설명 가능성 확보까지 직접 해보는 게 목적이었다.

### 무엇을 했나

담당 파트:
- **Object Detection**: MobileNetV2 ImageNet + COCO 클래스로 강아지 여부 먼저 판별
- **백본 비교 실험 → EfficientNetB0 선정**: MobileNetV2, MobileNetV3Large, EfficientNetB0 3개 직접 비교 후 최고 성능 모델 선정 (Stanford Dogs 120종)
- **단계적 성능 개선**: Validation set 구축 + epoch 확장 + EarlyStopping 적용
- **DALL-E 3 레시피 이미지 생성**: 조리 단계 키워드 파싱 → 최종 플레이팅 묘사로 변환 후 자동 생성
- **Grad-CAM**: 모델이 어떤 특징을 보고 판단하는지 Top-3 품종 히트맵 시각화

### 어떻게 해결했나

**분류 정확도 문제**: MobileNetV2 베이스라인(77.90%)의 한계 → 백본 비교 실험으로 EfficientNetB0 선정 후 Validation set 구축·epoch 확장·EarlyStopping 순서로 개선.

**모델 해석 문제**: 정확도 숫자만으로는 모델이 올바른 특징을 보는지 확인 불가 → Grad-CAM으로 히트맵 시각화, 실제로 귀·주둥이·털 패턴에 집중함을 확인.

### 성과

| Phase | 내용 | Top-1 Acc |
|-------|------|-----------|
| 베이스라인 | MobileNetV2 | 77.90% |
| 백본 교체 | EfficientNetB0 선정 | 85.52% |
| 최종 | Validation set + EarlyStopping | **86.99%** |

- 베이스라인 대비 **+9.09%p 성능 향상**
